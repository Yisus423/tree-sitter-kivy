# Design: Dynamic Class Rules

## Technical Approach

Extend the existing `class_rule` production in `grammar.js` with an optional `@ base` clause inside the angle brackets. Single rule type handles both simple `<Name>:` and dynamic `<Name@Base>:` class rules — purely additive, no backward-compatibility impact on existing CST trees.

## Architecture Decisions

### Decision 1: Extend `class_rule` vs. new `dynamic_class_rule`

| Option | Tradeoffs | Decision |
|--------|-----------|----------|
| **A: Extend `class_rule`** | Single rule type; existing `<Name>:` CST unchanged; downstream queries filter by `base` field presence | ✅ Chosen |
| **B: New `dynamic_class_rule`** | Clearer semantic separation but code duplication with `class_rule`; new node type means downstream tooling must match two rules instead of one | ❌ Rejected |

**Rationale**: A dynamic class IS a class rule — same `<...>:` construct with an optional inheritance clause. Extending is simpler, zero-risk for existing consumers, and naturally extensible to future `<A,B>:` and `<-Name>:` patterns.

### Decision 2: Multiple base fields via `repeat(seq('+', ...))`

Use tree-sitter's `repeat(seq('+', field('base', $.identifier)))` so each `+Base` produces a separate `base` field with the same name. Downstream tools see an array of `base` children—no custom logic needed.

### Decision 3: No scanner changes

`@` and `+` are regular ASCII characters. The external scanner (indentation tracking) is never invoked mid-line — only at line boundaries (NEWLINE, INDENT, DEDENT, BREAK). No structural change to the scanner's token set or logic.

### Decision 4: Keep `name` field required

`field('name', $.identifier)` remains required (non-optional). This means `<@Button>:` inserts a **MISSING** node for `name` (since `@` does not match `identifier = /[a-zA-Z_]\w*/`), then the optional `@ base` clause consumes `@Button` successfully. The parse tree contains a `class_rule` with `name: (MISSING)` and `base: (identifier)` — no top-level `(ERROR)` wrapper. Tree-sitter's error recovery represents the mistake structurally rather than as a silent fallthrough.

## Grammar Change

```js
class_rule: $ => seq(
    '<',
    field('name', $.identifier),
    optional(seq('@', field('base', $.identifier),
        repeat(seq('+', field('base', $.identifier))))),
    '>',
    ':',
    optional($._rule_body),
),
```

**Key properties**:
- `name` field always present (MISSING if absent in source)
- `base` field present only on dynamic class rules
- Multiple `+Base` → multiple `base` children (same field name, tree-sitter collects as array)

## Error Handling: `<@Button>:`

1. `<` enters `class_rule`
2. `identifier` vs `@` → no match → tree-sitter inserts **MISSING** node for `name`
3. `optional(seq('@', ...))` sees `@` → matches `Button` as `base`
4. `>` and `:` match normally
5. Result: `(class_rule name: (MISSING) base: (identifier))` — a structurally invalid class rule (no name), but parsed without cascading errors

This is correct behavior: the parser signals an error without discarding the rest of the rule body. Tooling receives a recognizable `class_rule` node with `base` and can report a targeted diagnostic ("dynamic class rule must have a name before `@`").
